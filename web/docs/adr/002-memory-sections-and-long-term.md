# ADR 002: メモリセクションと長期プロフィール

## 状態

採用（段階実装）

## 背景

「今日の事実」「長期プロフィール」「エントリ日の短期メモ」を同一テキスト列に混ぜると、モデルが記憶を当日ファクトとして扱うリスクがある。

## 決定

1. **プロンプト見出しを固定**
   - `## 今日の参照事実（structured）` … ToolFactCard 由来＋人間可読サマリ
   - `## 長期プロフィール（ゆっくり変わる前提）` … 長期記憶 DB 由来のみ
   - `## このエントリ日の短期メモ` … `MemoryShortTerm` のみ

2. **スキーマ（将来）**  
   `MemoryLongTerm.attributes` に `kind` 等を載せるか、正規化テーブル分割はデータ件数・クエリコストを見て別 ADR で判断。

3. **長期注入の検索（pgvector）**  
   `Embedding.targetType = MEMORY_LONG_TERM` で長期行を埋め込み、クエリは「ユーザー発言＋エントリ日」。ヒットが十分なければ **従来の impact 上位フォールバック**。

## 移行手順（段階）

1. **現段階（本リポジトリ）**: 見出し固定・短期 JSON メタ・長期は `MEMORY_LONG_TERM` 埋め込み＋類似検索フォールバック。`MemoryLongTerm.attributes` は `{}` のままで可。
2. **メタ付与が必要になったら**: `attributes` に `kind` / `validFrom` を追加し、既存行は `kind: "legacy"` 等で一括補填するマイグレーション用 SQL を用意（アプリ側では読み取りのみから開始）。
3. **クエリが複雑化したら**: `UserMemoryProfileFact` 相当の正規化テーブル案を再評価し、読み経路を段階的に切替（ADR 003 などで記録）。

## 結果

- 混線をプロンプト規約と検索経路の両方で抑制
- 埋め込み未整備ユーザーはフォールバックで従来どおり動作
