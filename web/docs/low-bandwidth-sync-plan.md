# 低速回線・データ上限対策 改善計画

## 1. 目標
- **200kbps級回線**でも主要操作（閲覧・入力・下書き保存）が待ち時間ストレス少なく使える。
- **月間データ上限到達後**でも「止まらない」体験（読み取り中心＋後送信）を維持する。
- **Wi‑Fi復帰時**に差分同期/画像再送を自動実行し、通常モードへ自然復帰する。
- 運用前提: アップロード上限は実質無制限だが、**写真は1日5枚まで**。

## 2. 全体アーキテクチャ（自動モード切替）

### 2.1 クライアント判定（ヒント）
- `navigator.connection`（`effectiveType`, `saveData`, downlink）
- 直近RTT/失敗率/タイムアウト率（アプリ計測）
- バッテリーセーブ状態（取得可能なら）

### 2.2 サーバー判定（最終決定）
- クライアントが送る `x-network-hint` を受け、サーバーで `normal / constrained / offline-priority` を決定。
- レスポンスヘッダーで `x-delivery-profile` を返却（クライアントUIにも反映）。
- 各APIで profile ごとに `fields`・`page size`・画像解像度を調整。

#### 判定ロジック（Wi‑Fi復帰の「両方判定」）
- **判定A: Wi‑Fi接続判定**
  - `navigator.connection?.type === "wifi"`（対応ブラウザのみ）
  - 取得できない環境では OS/UA 情報 + 直近速度から補完
- **判定B: 実効的に無制限回線か**
  - `saveData === false`
  - `effectiveType` が `4g` 以上
  - 直近 N 回の実測で `downlink`・RTT・失敗率が閾値を満たす
  - キャリア制限/テザリング疑い時は安全側で constrained 維持
- **昇格条件（normalへ戻す）**
  - A または B を満たし、かつ 2〜5 分連続で安定
- **降格条件（constrainedへ戻す）**
  - タイムアウト率・再送率・RTT悪化が閾値超過

### 2.3 モード定義
- `normal`: 従来どおり。
- `constrained`: 軽量JSON、画像縮小、バックグラウンド同期間隔を延長。
- `offline-priority`: 書き込みはローカルキュー優先、読み込みはキャッシュ優先。

## 3. 低速時UX（不快感を下げる設計）
- 画面上部に「軽量モード」バナー（解除不可・自動復帰型）。
- 画像はLQIP/サムネイル優先、原寸取得はタップ時のみ。
- 検索/一覧はページングを小さく（例: 10件）＋「続きを読み込む」。
- 地図・外部埋め込みは遅延読込（ユーザー操作で起動）。
- 同期ステータス表示（未送信件数、次回再試行時刻）で安心感を出す。

## 4. Wi‑Fi復帰時の自動追いつき

### 4.1 差分取得
- すべての一覧APIに `since` と `cursor` を導入。
- クライアントは最終同期トークン（`lastSyncedAt` + `cursor`）を保持。
- Wi‑Fi復帰時は「差分のみ取得 → 競合解決 → キャッシュ更新」。

### 4.2 差分送信
- ローカル変更を**操作ログ（oplog）**として保存。
- `POST /sync/changes` へバッチ送信（冪等キー付き）。
- 画像は参照付きメタデータ先行送信、実体はWi‑Fiで遅延アップロード。
- 写真は1日5枚制限をサーバー側で検証し、超過時は翌日キューへ自動繰越。

### 4.3 競合解決
- まずは `updatedAt` + フィールド単位マージ。
- 将来的には CRDT も検討（テキスト編集が増える場合）。

## 5. ローカル保存（可能か？への回答）
**可能です。** ただし保存先を分けるのが重要です。

- 構造化データ（エントリ、設定、同期トークン）: `IndexedDB`
- 画像バイナリ: `Cache Storage` または `IndexedDB Blob`
- 同期キュー（未送信変更）: `IndexedDB`（永続化）
- 暗号化: 端末内機密度に応じて at-rest 暗号化を検討

推奨方針:
1) 「必須データは必ずローカル保持」
2) 「サーバーは最終的整合性」
3) 「通信可能時に追いつく」

### 欠損不可データ優先順位（確定）
1. **日記本文**
2. **写真圧縮データ**

## 6. 具体施策（追加要望を反映）

### 6.1 画像アップロード圧縮（モバイル時のみ強制）
- プロファイル:
  - `constrained`: 長辺1280px, quality 0.6
  - `offline-priority`: 長辺960px, quality 0.5
- EXIF回転補正後に圧縮、元画像ハッシュを保持。
- Wi‑Fi時に高画質再アップロードを許容する二段階方式。

#### Google Photos Picker 最適化（質問への回答）
**可能**。通信量最小化のため、以下の方式を推奨:
- クライアントは `mediaItemId` のみサーバーへ送信（本体バイナリは原則送らない）。
- サーバーが Google API から直接取得し、サーバー側で圧縮・派生画像生成。
- クライアントには用途別サイズ（thumb / feed / detail）のみ返却。
- 必要時のみ高解像度を遅延配信。
- これにより、**ユーザー端末↔自サーバー間**の転送量を大幅削減できる。

### 6.2 API差分取得 + ページング
- `GET /entries?since=...&cursor=...&limit=...`
- レスポンスを summary/detail に分割。
- 初回は summary のみ、詳細は遅延取得。
- 優先画面は **Today / Calendar** を先行対象にする。

### 6.3 prefetch抑制
- `next/link` は主要導線のみ prefetch on。
- 低速モードでは prefetch off（動的切替）。

### 6.4 軽量モードバナー + 同期頻度制御
- ポーリング間隔延長（例: 15s → 120s）
- push通知利用時はpull最小化。
- 低速時でも Today/Calendar は先読みを継続（ただし summary のみ・低優先度で実行）。

## 7. 実装フェーズ

### Phase 0（1〜2週）: 可視化
- 通信量、APIサイズ、画像サイズ、同期失敗率を計測。
- ネットワークプロファイル別ダッシュボード。

### Phase 1（2〜3週）: 体感改善
- 軽量モードUI、prefetch抑制、画像圧縮、地図遅延読込。

### Phase 2（3〜4週）: 同期基盤
- `since/cursor`、oplog、バッチ送信、再送制御。

### Phase 3（2〜3週）: Wi‑Fi追いつき最適化
- 背景同期、画像再アップロード、競合解決改善。

## 8. 受け入れ基準（KPI）
- 初回表示までの通信量: **-60%以上**
- 200kbps時の主要画面操作成功率: **95%以上**
- オフライン/低速中の入力消失: **0件**
- Wi‑Fi復帰後5分以内の同期完了率: **95%以上**

## 9. リスクと対策
- 端末ストレージ不足: LRU削除と上限管理（例: 300MB）
- 競合増加: 編集単位の縮小、衝突UI提示
- 実装複雑化: feature flagで段階投入

## 10. 確定した前提
1. アップロード総量は実質無制限、ただし写真は1日5枚まで。
2. 欠損不可の優先は「写真圧縮データ」「日記本文」。
3. 低速時に最優先で快適化する画面は Today / Calendar。
4. 端末ローカル保存期間の法務・セキュリティ制約はなし。
5. Wi‑Fi判定は「Wi‑Fi接続判定」と「実効無制限回線判定」の両方を使う。

## 11. 追加で確認したい点（実装前）
1. 「写真圧縮データ」と「日記本文」のどちらを優先保護するか（同時障害時の優先順位）。
2. Google Photos 連携で、サーバー側代理取得に必要な同意文言/規約更新は実施可能か。
3. 写真1日5枚制限の基準時刻（ユーザー現地時間の0時リセットでよいか）。
4. Today/Calendar の「最低限表示」に必要な項目定義（例: タイトル・時刻・感情ラベルのみ）。

## 12. 改善開始タスク（今回着手）

### 12.1 写真1日5枚制限のUX改善
- 常時表示項目を追加する:
  - 「本日の残り枚数（例: あと3枚）」
  - 「次回リセット時刻（ユーザー現地時間）」
- 表示位置:
  - Today画面の投稿アクション付近
  - Calendar画面の新規作成導線付近
- API案:
  - `GET /api/photos/quota` → `{ remaining, dailyLimit, resetAt }`
- 受け入れ基準:
  - 投稿前に残数が確認できる
  - 上限到達時の理由と再開時刻が明確

### 12.2 Today / Calendar 最低限表示項目の固定
- 低速モードで最初に返す `summary` 項目を固定:
  - 共通: `id`, `date`, `title`, `updatedAt`
  - Today: `weatherSummary`, `hasPhoto`, `moodLabel`
  - Calendar: `startAt`, `endAt`, `category`, `isAllDay`
- 詳細項目（本文、高解像度画像、補足メタ）は遅延取得へ分離。
- 受け入れ基準:
  - 200kbps相当で Today / Calendar の初回描画を優先
  - 詳細未取得でも画面骨格と主要操作が成立

### 12.3 Google Photos 代理取得の同意・規約整備
- 実装前に以下を更新:
  - 利用規約: サーバーが第三者APIから写真を代理取得・圧縮する旨
  - プライバシーポリシー: 取得対象、保存期間、派生画像生成の説明
  - 同意UI: 初回連携時に明示同意チェック
- 受け入れ基準:
  - 同意ログ（timestamp / policy version）を監査可能
  - 規約未同意ユーザーには代理取得フローを無効化
